🚀 Quick Start

1 Installation

Activate the collection conda environment before connecting any devices or running scripts.

source ~/miniconda3/bin/activate
conda activate vb

2 Data Collection

The collection pipeline is organized around Quest tracking, packed camera streams and a task config YAML. Follow these steps in order.

Step 1 — Confirm Quest connection

Check that the device state is device, not unauthorized, offline or no permissions.

adb devices

Step 2 — Forward ports

adb forward tcp:7777 tcp:7777
adb forward tcp:50010 tcp:50010
adb forward --list

Step 3 — Prepare the Quest app

  • Open the tracking app (vb-tcp) inside the headset.
  • Select Y: TCP Mode.
  • Keep the app in the foreground — do not let the Quest sleep.

Step 4 — Verify camera devices

Typical packed-camera mapping: /dev/video0 = right hand packed, /dev/video2 = left hand packed. 
ls /dev/video*
v4l2-ctl --list-devices

Step 5 — Edit the task config

Config path: VB-VLA/Data_collection/config/VB_task_config.yaml 

task:
  type: single          # single or bimanual
  name: raw_0321_data5
  copy_raw_data: true
  single_hand_side: left  # left or right (only used when type is 'single')

recorder:
  output: "./data"
  camera:
    format: "MJPG"
    width: 3840
    height: 800
    auto_exposure: 3      # 1=auto, 3=manual
    exposure: 300         # manual shutter (µs), only active when auto_exposure=3
    auto_white_balance: 0 # 0=manual, 1=auto
    wb_temperature: 4600  # manual white balance (K), only active when auto_white_balance=0
    brightness: 0
    gain: 40
    gamma: 50

  camera_paths:
    left_hand: "/dev/video2"
    # right_hand: "/dev/video0"

Step 6 — Launch the collection script

python VB-VLA/Data_collection/vitamin_b_data_collection_pipeline/00_get_data.py \
  --cfg VB-VLA/Data_collection/config/VB_task_config.yaml

3 Data Processing

1. Merge raw demos

python Data_collection/vitamin_b_data_collection_pipeline/merge_raw_demos.py \
  --source data/raw_data0 data/raw_data1 \
  --target data/raw_merged

2. Update dataset naming in config.py

# Dataset naming — edit these three fields together
DATASET_TRAIN_NAME: str = "0118_data_1smooth"   # used for repo_id, asset_id, exp_name
DATASET_RAW_TASK_NAME: str = "raw_0118_data"    # raw data path: data/{task_name}/
DATASET_REPO_NAMESPACE: str = "your-namespace"  # HuggingFace namespace

3. Choose task mode

Four settings control single-arm vs bimanual behaviour — keep them consistent across preprocessing and training:

Single-Arm

  • task_type: "single"
  • single_hand_side: "left" or "right"
  • single_arm: True
  • no_state: False → state_dim=7, action_dim=10
  • Training config: pi05_single / pi05_single_vitac

Bimanual

  • task_type: "bimanual"
  • single_arm: False
  • no_state: False → state_dim=20, action_dim=20
  • Training config: pi05_bi
Setting no_state: True keeps only gripper width in the state vector. If you use this, make sure to update --state_dim / --action_dim in your training config accordingly.

4. One-click conversion to LeRobot format

python Data_collection/run_data_collection_pipeline_le.py

5. Upload to Hugging Face

# Authenticate (enter your token when prompted)
hf auth login

# Use the China mirror if needed
export HF_ENDPOINT=https://hf-mirror.com

# Upload the processed dataset
hf upload your-namespace/your-dataset /path/to/processed_dataset . \
  --repo-type dataset
UMI gripper labeling files: Google Drive  ·  Foot pedal setup software: Download

4 Visualization

Use the rerun-based visualization tool to inspect collected datasets, replay robot trajectories and verify that camera, state and action streams are aligned.

Setup

conda create -n vbviz python=3.11 -y
conda activate vbviz
# install visualization dependencies in ~/VB-VLA/robot_visualization-main

Usage

# Single episode
python src/viz_rerun.py /path/to/dataset -e 0

# Multiple non-consecutive episodes
python src/viz_rerun.py /path/to/dataset -e 0 3 7

# A range of episodes
python src/viz_rerun.py /path/to/dataset -e 0-10

# Mixed range and individual episodes
python src/viz_rerun.py /path/to/dataset -e 0-5 8 12-15

# Save a .rrd replay file and open it later
python src/viz_rerun.py /path/to/dataset --save output.rrd
rerun output.rrd

Source: robot_visualization repository

🤖 Robot Setup

Hand-Eye Calibration

Run the robot API and the Quest-to-end-effector pose capture script in two separate terminals, then save calibration sequences for solving and validation. 

# Terminal 1 — start the robot API
python3 api_demo.py

# Terminal 2 — capture calibration poses
conda activate vb
cd ~/VB-VLA

python tools/cali_hand_eye/get_quest_ee_pose.py \
  --side left \
  --no-auto \
  --passive \
  --save-dir /path/to/calibration_run

Collection Tips

  • Use small translations and small rotations; keep translation smaller than rotation.
  • Keep the controller's black face pointing toward the Quest.
  • Stay near the center of the Quest field of view — avoid the edges.

Validation Tips

  • Record dual timestamps — Quest packet timestamp, local receive time, and EE read time — to diagnose latency jitter.
  • Static test: hold both robot and hand still for 20 frames. If Quest pose still jumps (>5 mm / >1°), the tracking or timing has issues.
  • Check the robot coordinate frame, the arm root frame, and the transform between them (S matrix correctness).
  • Verify forward kinematics by moving the gripper and measuring the displacement with a ruler.
Hardware reference: ViTaMIn-B project page

Robot API

The hardware interface exposes a compact state-machine API for reading robot state, sending joint-space targets and switching modes during recording or playback.

Endpoints

Method Endpoint Description
GET /state Read state-machine status and hardware status
POST /action Send left/right arm joint targets and gripper targets
POST /command Switch modes; start/stop recording or playback
GET /error Read current error state

Action Body

body = {
    LEFT_ARM:      left_arm_target_positions,   # 7 joint angles in radians
    LEFT_GRIPPER:  left_gripper_target_position, # width in meters
    RIGHT_ARM:     right_arm_target_positions,   # 7 joint angles in radians
    RIGHT_GRIPPER: right_gripper_target_position # width in meters
}

Action Space Notes

VLA policies typically output end-effector deltas (Δposition, Δrotation, gripper action), while the API accepts joint-space targets. An action adapter is required:

  • Forward kinematics (FK) to get the current end-effector pose.
  • Apply Δpose to compute the target end-effector pose.
  • Inverse kinematics (IK) to convert to joint targets.
  • Apply joint limits, interpolation and filtering before sending.

📮 Submission

Once you are satisfied with your model's performance, submit your results through the submission form. The submitted files include:

  • A README (.md) file providing detailed instructions on how we can evaluate your policy.
  • A Docker image URL or HuggingFace repo link containing your checkpoints, source code and all dependencies required for evaluation.
  • A .txt file containing the rollout results you performed locally.
  • Source code link (optional, but highly encouraged). This will NOT be used for evaluation, but helps us debug issues if anything goes wrong during evaluation.
Please check the submission form for more detailed requirements and deadlines.
